# MusiCue v0.2a — Frame-first metadata

**Goal:** Make every event in `analysis.json` and `cuesheet.json` carry frame numbers and SMPTE timecode in addition to `t` (seconds), so animation, VFX, and editorial pipelines can pull MusiCue data without doing their own time-to-frame math.

**Why now:** Today every exporter that needs frames computes them at export time, with the project's FPS as a per-export option. That works for headline cases but means: (a) the FPS chosen at compile time is invisible at export time; (b) downstream tools that round-trip the cuesheet (e.g., Resolve marker imports) lose the frame field; (c) tools that read `analysis.json` directly get nothing. Promoting frame + timecode to first-class fields collapses three classes of bug.

**Scope rule:** This is a schema bump and a tiny CLI / API extension. No new exporters, no new analysis. The video / audio export tabs in the UI stay deferred. Editorial-format exporters (EDL/FCPXML/etc.) are a separate spec (`v0.2b-editorial-exports.md`) that *depends on this one shipping first*.

**Acceptance test:**

> `musicue analyze song.wav --fps 24 --out runs/song/`. Open `runs/song/analysis.json`. Every `BeatEvent`, `OnsetEvent`, `SectionEvent`, `SectionTransition`, `PhraseEvent`, and `MidiNote` has `frame` (int) and `timecode` (string `HH:MM:SS:FF`) alongside its existing `t` field. `analysis_config.fps` records `24`. Then `musicue compile runs/song/analysis.json --grammar concert_visuals --fps 30 --out cs.json` produces a cuesheet whose `events[].frame` and `events[].timecode` are *recomputed* at 30 fps (not blindly carried over from the analysis FPS). `cs.json.fps` records `30`. Round-tripping by re-loading `cs.json` and re-exporting at `--fps 30` gives identical frame numbers.

---

## Decisions locked

| Decision | Choice | Why |
|---|---|---|
| Source of truth | `t` (seconds, float) stays canonical. `frame` and `timecode` are derived. | Avoids drift across re-encodes and re-FPS jobs. |
| Drop-frame timecode | Support 23.976/29.97 drop-frame and 24/25/30 non-drop. Default non-drop. | Editorial users need DF; everyone else doesn't care. |
| Where FPS lives | `analysis_config.fps` and `cuesheet.fps` (top-level). Defaults to 24. | Two fields because compile can override (a 24p analysis can be re-framed at 30p for a different target). |
| Existing analyses | Backwards-compatible: if `frame`/`timecode` missing, exporters compute on-the-fly using `cuesheet.fps`. New CLI runs always populate them. | No forced re-analyze of existing libraries. |
| Schema version | Bump `analysis.json` to `1.2`, `cuesheet.json` to `1.2`. | Frame fields are additive; old consumers ignore them. |
| API exposure | `/api/songs/{id}/analyses/{aid}/export` accepts `fps` (already does for AE/disguise). Now also accepted for *every* format and stamped onto every event in the produced file. | Trivial extension. UI's Export modal grows an FPS field on the always-shown row. |

## §1. Schema additions

### `musicue/schemas.py`

Add a mixin or helper module:

```python
# musicue/timecode.py
from typing import Literal

DropFrame = Literal["23.976", "29.97"]

def t_to_frame(t: float, fps: float) -> int:
    return int(round(t * fps))

def t_to_timecode(t: float, fps: float, drop_frame: bool = False) -> str:
    """SMPTE HH:MM:SS:FF (semicolon-separated final field for DF)."""
    # Implement standard DF math (1 dropped per minute except every 10th)
    # See https://andrewduncan.net/timecodes/ for the canonical formula.
    ...
```

Add to every event model:

```python
frame: int | None = None
timecode: str | None = None
```

Add to `AnalysisConfig`:

```python
fps: float = 24.0
drop_frame: bool = False
```

Add to `CueSheet`:

```python
fps: float = 24.0
drop_frame: bool = False
```

### Population strategy

- **At analyze time:** `pipeline.run_analysis(audio, cfg)` reads `cfg.fps` and stamps `frame` + `timecode` on every event before serializing.
- **At compile time:** `compile_analysis(analysis, grammar, fps=...)` accepts a new `fps` arg. If provided, recomputes frames on every event in the resulting cuesheet (overrides analysis fps). Otherwise inherits from analysis.
- **At export time:** Existing per-exporter `fps` opts (AE, disguise) become *display* concerns only — when set, they call `t_to_frame` themselves on already-computed `t`. Cuesheet's `fps` is the canonical animation rate.

## §2. CLI changes

```bash
# Analyze
musicue analyze song.wav --fps 24                      # NEW flag
musicue analyze song.wav --fps 23.976 --drop-frame     # NEW flag

# Compile
musicue compile analysis.json --grammar X --fps 30     # NEW flag, overrides

# Render
musicue render song.wav --fps 24 --target after_effects --out cs.jsx
```

`--fps` defaults to 24. If both analyze and compile specify FPS, compile wins (compile is closer to the export target).

---

## §3. UI changes

### Library page

No change.

### Editor → Export modal

Top row gains an **FPS** input that's *always* visible (not just for AE/disguise). Default 24. The FPS the user types here becomes the cuesheet's animation rate; downstream exporters use it.

### `lib/exportApi.ts`

`ExportRequest.fps` becomes required-with-default rather than format-conditional. The backend always recomputes frames at the supplied FPS before writing the file.

---

## §4. Exporter touch-ups (minimal)

Every exporter gets ~3 lines: write the cuesheet's `fps` into a header comment / metadata field where one exists, and use `event.frame` directly when available. Specifically:

- **csv** — add `frame_number` column.
- **json** — already serializes events as dicts; nothing to do.
- **midi** — no frame concept; ignore.
- **after_effects** — use `frame` directly instead of recomputing.
- **touchdesigner** — add `frame` column to the events CSV.
- **osc** — add `"frame"` field next to `"t"`.
- **houdini** — already FPS-aware; switch to `frame` field.
- **disguise** — recompute timecode using cuesheet's FPS at write time.
- **unreal** — add `frame` to event tracks.

---

## §5. Tests

- **Unit:** `t_to_frame`, `t_to_timecode` round-trip and DF math (use the SMPTE reference values from the andrewduncan.net page).
- **Integration:** `analyze --fps 24` then `compile --fps 30` yields different frames per event; verify the math with a hand-computed expected.
- **API:** POST to export with `fps=24` vs `fps=30` returns CSVs whose `frame_number` column differs accordingly.
- **Backwards-compat:** load a v1.1 `analysis.json` (no frame fields), run `compile --fps 24`, verify exporter output still works.

---

## §6. Out of scope

- Editorial markers (EDL/FCPXML/etc.) — `v0.2b-editorial-exports.md`.
- Variable-FPS / mixed-rate timelines (a 24p timeline with a 60p insert).
- Sub-frame precision for OSC / MIDI — those keep using `t`.
- Per-event FPS override — a single rate per cuesheet is enough.

---

## §7. Risks

- **Drop-frame math** is a famous source of off-by-one bugs. Use a tested reference implementation; do *not* reimplement from memory.
- **Existing `runs/` dirs** with v1.1 analyses keep working but lose the frame fields; users who want them re-run analyze. Document this in the changelog.
