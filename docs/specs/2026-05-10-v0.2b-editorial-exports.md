# MusiCue v0.2b — Editorial marker exports

**Goal:** Ship four new exporters that drop straight into editorial pipelines: **EDL** (legacy/Avid), **Final Cut Pro X XML** (FCPXML, used by Resolve and FCPX), **Premiere Pro marker XML**, and **DaVinci Resolve marker CSV**. Each emits MusiCue's section boundaries and key impulse events as visible markers/clips on a timeline so editors can cut to music without ever opening a separate analysis tool.

**Why now:** MusiCue's existing exporters target motion graphics (AE), real-time visuals (TD/OSC/Unreal), and lighting (disguise). Editors — the people who actually cut music videos, trailers, and promo content — get nothing today. The data is already in `analysis.json`. This is plumbing, not research.

**Depends on:** `v0.2a-frame-timecode.md` shipping first. EDL and FCPXML both require frame-accurate SMPTE timecode at the cuesheet level. Without v0.2a, every exporter would have to reimplement timecode math.

**Acceptance test:**

> `musicue export cs.json --target edl --fps 24 --out song.edl` produces a valid CMX 3600 EDL that opens in Avid, Resolve, and Premiere and displays one marker per detected section + per detected drop. `--target fcpxml` produces a valid FCP X library XML that imports into Resolve 18+ and FCPX 10.6+ with markers visible on a placeholder clip. `--target premiere_markers` produces a marker CSV that Premiere's Marker Importer extension accepts. `--target resolve_markers` produces a CSV that Resolve's Marker → Import accepts. All four formats round-trip through their target tool without timecode drift.

## Decisions locked

| Decision | Choice | Why |
|---|---|---|
| Marker source events | Sections (always), section transitions (always), drum onsets where `drum_class == 'kick'` AND on a downbeat (configurable), CLAP labels with `score >= 0.7` (configurable) | Editors care about song structure, hits, and named moments. Per-beat / per-onset markers would flood the timeline. |
| Marker color convention | Sections → blue, transitions → red, drops/CLAP → yellow, kicks → green | Each format encodes color differently; fall back to "default" where the format doesn't support color. |
| Marker name format | `<label>` for sections; `<from> → <to>` for transitions; `<drum_class>` or `CLAP: <label>` for events | Readable in the timeline panel, machine-parseable. |
| Single-clip vs multi-clip in FCPXML | Single placeholder clip with markers attached | Editors want to drop these into an existing edit, not have MusiCue dictate clip structure. |
| MusiCue audio inclusion | Optional `--include-audio` flag for FCPXML / Premiere XML; embeds the source audio as a clip with markers attached. Off by default. | Bigger files, but useful when starting an edit from scratch. |
| EDL flavor | CMX 3600 (the universal one). Reels follow `MUSICUE` convention. | CMX 3600 reads everywhere; AAF / OMF are out of scope. |

## §1. New exporters

```
musicue/exporters/
  edl.py
  fcpxml.py
  premiere_markers.py
  resolve_markers.py
```

Each follows the existing exporter signature:

```python
def export(cuesheet: CueSheet, out_path: Path, **opts) -> None:
```

Each accepts:

- `fps: float` (inherited from cuesheet by default)
- `drop_frame: bool` (inherited)
- `marker_sources: set[str] = {"sections", "transitions", "drops"}` — what to emit
- `min_clap_score: float = 0.7`
- `include_audio: bool = False` (FCPXML / Premiere only; embeds source audio path)

### EDL (CMX 3600)

A line-based text format. Each event is a 1-frame "marker" event. Spec ref: https://documentation.apple.com/en/finalcutpro/usermanual/index.html#chapter=96%26section=1 (also countless community summaries — the format is ~50 years old and well-documented).

```
TITLE: Song title — MusiCue auto markers
FCM: NON-DROP FRAME

001  MUSICUE  V     C        00:00:00:00 00:00:00:01 00:00:00:00 00:00:00:01
* FROM CLIP NAME: section_intro
* COLOR: blue
* COMMENT: intro

002  MUSICUE  V     C        00:00:08:14 00:00:08:15 00:00:08:14 00:00:08:15
* FROM CLIP NAME: section_verse
* COLOR: blue
* COMMENT: verse
```

### FCPXML 1.10

XML format. Skeleton:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE fcpxml>
<fcpxml version="1.10">
  <resources>
    <format id="r1" frameDuration="1/24s" width="1920" height="1080"/>
    <asset id="r2" name="song" duration="<total>s" format="r1" hasAudio="1" hasVideo="0"/>
  </resources>
  <library>
    <event name="MusiCue markers">
      <project name="<song title>">
        <sequence format="r1" duration="<total>s" tcStart="0s">
          <spine>
            <asset-clip ref="r2" offset="0s" duration="<total>s" name="<song>">
              <marker start="8.583s" duration="1/24s" value="verse"/>
              <marker start="32.5s" duration="1/24s" value="chorus" note="ramp: ease_in_out"/>
              ...
            </asset-clip>
          </spine>
        </sequence>
      </project>
    </event>
  </library>
</fcpxml>
```

Marker attributes the format supports: `start`, `duration`, `value` (name), `note`, `completed` (for to-do markers), `priority`. Color is encoded via marker name prefix in some Resolve versions; we'll emit prefixed names (`[blue]verse`) for compatibility.

### Premiere marker CSV

Premiere's stock CSV importer expects:

```
Marker Name,Description,In,Out,Duration,Marker Type
intro,Section intro,00:00:00:00,00:00:00:00,00:00:00:00,Comment
verse,Section verse,00:00:08:14,00:00:08:14,00:00:00:00,Comment
chorus,Ramp: ease_in_out,00:00:32:12,00:00:32:12,00:00:00:00,Comment
```

### Resolve marker CSV

Resolve uses a different column set:

```
#,Source In,Source Out,Track,Type,Note,Color,Source File
1,00:00:00:00,00:00:00:00,V1,Section,intro,Blue,
2,00:00:08:14,00:00:08:14,V1,Section,verse,Blue,
3,00:00:32:12,00:00:32:12,V1,Transition,verse → chorus,Red,
```

## §2. CLI / API additions

```bash
musicue export cs.json --target edl --fps 24 --out song.edl
musicue export cs.json --target fcpxml --include-audio --out song.fcpxml
musicue export cs.json --target premiere_markers --min-clap-score 0.8 --out markers.csv
musicue export cs.json --target resolve_markers --out resolve.csv
```

`--marker-sources sections,transitions,drops` (comma-separated) for fine control.

UI's Export modal grows the four new entries in the Format dropdown and a small "Marker sources" multi-select that appears when one of these formats is picked.

## §4. Tests

- **Format compliance:** snapshot tests for each format against fixture cuesheets. Diff the generated output against committed expected files.
- **Round-trip:** for FCPXML, parse our output back with `xml.etree.ElementTree`, count markers, verify timestamps match cuesheet events.
- **Live tools:** *not* automated. Manual UAT in: Resolve 18, Premiere 2024, FCPX 10.7, Avid Media Composer 2023. Documented as a release-time checklist.
- **Drop-frame:** every exporter gets a 29.97 DF test that verifies the timecode strings use semicolons for the final field (DF convention).

---

## §5. Out of scope

- AAF / OMF / Avid-native binary formats (heavy, niche).
- Audio Edits Embed (AAE) for Premiere chunks.
- Source-clip-aware editing (assumes a single placeholder clip per song).
- Multi-track timelines (one track per stem).
- Round-tripping edits *back into* MusiCue (a v0.4 collab feature).

---

## §6. Risks

- **Format-spec drift:** EDL and FCPXML have multiple historical versions. We commit to CMX 3600 for EDL and FCPXML 1.10 for FCPX/Resolve, both of which have been stable for years. If users ask for FCPXML 1.11, treat as additive.
- **Marker color encoding** is the most format-divergent piece. Document the per-format mapping; if a tool ignores our color attribute, surface "use the marker name prefix" as a workaround.
- **Resolve CSV import** is finicky about line endings and BOM. Write CRLF + UTF-8-with-BOM specifically for that format.
