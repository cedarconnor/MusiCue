# MusiCue v0.2c — Beat patterns and pattern-aware grammar

**Goal:** Detect repeating rhythmic patterns (4-bar phrase blocks, drum fills, syncopation islands, "every Nth beat" structures) and expose them as grammar selectors so authors can write rules like *"only fire on bar 1 of every 8-bar phrase"* or *"emphasize the snare hit two bars before each chorus"*. Moves MusiCue's grammar layer from "every beat does X" to actually-musical phrasing.

**Why now:** BeatEvents already carry `bar` and `beat_in_bar` (populated by All-In-One). Section transitions are already detected. We have all the inputs for pattern mining; we're just not doing it. Without pattern selectors, authors who want musical-feel grammars resort to copying onset filters around manually.

**Scope rule:** Pure analysis layer. No new exporters, no new ML models, no new dependencies. Pattern detection is heuristic over already-detected events. The grammar selectors are new filter primitives in the existing YAML grammar engine.

**Acceptance test:**

> A song with detected sections and beats. After analysis, `analysis.json` contains a new `patterns` block with: `phrases` (4 / 8 / 16-bar repeat blocks), `fills` (drum onset density spikes near downbeats), `syncopation` (per-bar score 0.0-1.0). A grammar can write `filter: "phrase_position == 1 and phrase_length == 8"` to fire only on the first bar of each 8-bar phrase. Compiling that grammar against the song produces a track whose events fall on those exact bars (verifiable by inspecting the cuesheet). A second grammar with `filter: "is_fill()"` produces a track whose events cluster at the end of phrase blocks where drum density spikes.

---

## Decisions locked

| Decision | Choice | Why |
|---|---|---|
| Pattern types in v0.2c | Phrase blocks (bar-period repetition), drum fills, syncopation, downbeat strength | Covers ~80% of "musical phrasing" usage. More exotic patterns (polyrhythm, hemiola) deferred. |
| Phrase detection algorithm | Greedy bar-window correlation: window the per-bar onset histogram, find the dominant period (4 / 8 / 16) by autocorrelation, label each bar with `phrase_id` and `phrase_position`. | Cheap, interpretable. Real music-theoretic phrasing is too heterogeneous for a single-pass heuristic; users can override per-section. |
| Fill detection | Per-bar drum onset count z-score; flag bars where score > 1.5 AND bar position is N-1 of an N-bar phrase. | Captures the "fill before the next phrase" feel without needing a trained model. |
| Syncopation score | Sum of off-beat onset strengths divided by sum of all onset strengths, per bar. | Standard MIR feature; cheap; lines up with how musicians describe songs. |
| Grammar surface | New filter primitives. No new track types. | Backwards-compatible. Existing grammars keep working. |
| Schema bump | `analysis.json` → 1.3. New `patterns` top-level block. | Additive. |

### `musicue/analysis/patterns.py` (new)

```python
def detect_patterns(beats: list[BeatEvent], onsets: dict[str, list[OnsetEvent]],
                    sections: list[SectionEvent]) -> Patterns:
    """Pure function: takes event lists, returns a Patterns object.
    Runs in milliseconds; called at the end of run_analysis()."""
```

Output:

```python
class PhraseBlock(BaseModel):
    bar_start: int
    bar_end: int        # exclusive
    length: int         # 4, 8, 16, etc.
    section_label: str  # whichever section this phrase is in
    confidence: float   # how strong the period correlation was

class FillEvent(BaseModel):
    bar: int            # which bar contains the fill
    t_start: float
    t_end: float
    density_zscore: float
    leads_into: str | None  # next section label, if a transition is near

class Patterns(BaseModel):
    phrases: list[PhraseBlock]
    fills: list[FillEvent]
    syncopation_per_bar: list[float]   # one score per bar
    bar_count: int
```

### `AnalysisResult` schema

```python
class AnalysisResult(BaseModel):
    schema_version: str = "1.3"
    ...
    patterns: Patterns | None = None
```

### Per-beat enrichment

Existing `BeatEvent` already has `bar` and `beat_in_bar`. We *don't* mutate them. Instead, every beat gains a derived field at compile time:

```python
class BeatEvent(BaseModel):
    ...
    phrase_id: int | None = None      # index into Patterns.phrases
    phrase_position: int | None = None # 1-indexed bar within phrase
    phrase_length: int | None = None  # convenience copy
    is_fill: bool = False
    syncopation: float | None = None  # bar-level score
```

These fields stay `None` for songs analyzed pre-v0.2c (back-compat: all selectors that reference them treat None as "no match").

## §2. Grammar selectors (new)

### Filter functions

Add to the filter expression language (`musicue/compile/scoring.evaluate_filter`):

```yaml
# Beats only
filter: "phrase_position == 1"            # downbeat of every phrase
filter: "phrase_position == 1 and phrase_length == 8"   # first bar of 8-bar
filter: "phrase_position == phrase_length"  # last bar of every phrase
filter: "phrase_position == phrase_length and bar_to_section_end <= 2"
                                          # last bar before section change
filter: "is_fill()"                       # drum density fills
filter: "syncopation > 0.4"               # syncopated bars
filter: "every_nth(4)"                    # every 4th beat from song start
filter: "every_nth(8, offset=0)"          # bars 0, 8, 16, ...
```

### Score multipliers

Patterns can also feed scoring without being hard filters:

```yaml
score:
  base: "strength"
  multipliers:
    - { when: "phrase_position == 1", factor: 1.5 }
    - { when: "is_fill()", factor: 2.0 }
    - { when: "syncopation > 0.4", factor: 1.3 }
```

Existing score-multiplier syntax stays unchanged; the `when` expressions just have new vocabulary available.

### Helper functions exposed in filter context

| Function | Returns |
|---|---|
| `phrase_position` | int (1..phrase_length) or None |
| `phrase_length` | int (4 / 8 / 16) or None |
| `phrase_id` | int or None |
| `bar_to_section_end` | int (bars remaining in current section) |
| `is_fill()` | bool |
| `is_phrase_start()` | shorthand for `phrase_position == 1` |
| `is_phrase_end()` | shorthand for `phrase_position == phrase_length` |
| `syncopation` | float (per-bar score) |
| `every_nth(n, offset=0)` | bool |

## §3. Built-in grammar updates

Each shipped grammar gains *one* pattern-aware track to demonstrate the feature:

- **concert_visuals** — `phrase_pulse` track: impulse on every phrase start.
- **lighting** — `fill_flash` track: bright impulse during fills.
- **camera_edit** — strengthen section-cut weights when the cut would land on a phrase boundary.
- **character_animation** — `phrase_breath` envelope: longer release envelopes at phrase ends.

Adding these is a single grammar diff per file; ~10 LOC each.

## §4. Tests

- **Unit (`tests/test_patterns.py`):** synthetic 16-bar input with a known 4-bar repeat. Verify `phrases` finds 4 blocks of length 4. Verify edge cases: 8 bars (two 4-bar phrases? one 8-bar phrase? choose the higher-correlation interpretation), 12 bars (no clean factor — return one 12-bar phrase with low confidence).
- **Unit (`tests/test_patterns_fills.py`):** craft an onset stream with a clear density spike in the bar before a section change. Verify `fills` includes that bar.
- **Integration:** add a fixture song to `tests/fixtures/` with hand-authored expected `phrases`, run end-to-end, snapshot the output.
- **Grammar:** unit tests for each new filter primitive (`phrase_position`, `is_fill()`, `every_nth()`).
- **Backwards-compat:** load an old (v1.2) `analysis.json` lacking `patterns`; compile the same grammar; verify no crashes — pattern-aware filters silently no-op.

## §5. UI implications (deferred)

The Editor doesn't need to surface patterns in v0.2c — they live in the data and the grammar consumes them. A nice future v0.3 polish: show phrase-block boundaries as faint vertical lines on the mix lane, and shade fill bars. Out of scope here.

---

## §6. Out of scope

- Genre-specific pattern templates (e.g., "EDM 8-bar buildup → drop").
- Polyrhythm / cross-rhythm detection.
- Section-relative phrasing (4-bar phrases inside one verse, 8-bar inside the chorus). Single-pass for now.
- Manual phrase override in the Editor — nice but waits until the editor's edit-layer ships.
- ML-trained pattern detection (this is heuristic; a learned model is research scope).

---

## §7. Risks

- **Phrase ambiguity:** real music has 4-bar phrases that are *also* 8-bar phrases nested. We pick one (highest autocorrelation peak); document that ambiguity in the changelog.
- **Free-time / rubato sections:** our heuristic will give low-confidence phrases or none. That's fine — filters return None and pattern-aware tracks fall back to nothing for that section.
- **Bar-counting off-by-one:** All-In-One's bar numbers are 1-indexed in some configurations and 0-indexed in others. Pin the convention in the schema and convert on ingest.
